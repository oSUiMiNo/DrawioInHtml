新バージョンを公開する（最頻出）

既に PAT を持っているなら 5 分で公開できる。

標準フロー（コード変更があるとき）

  1. コードを変更してテスト 
    cd /home/osuim/Dev/DrawioInHtml
npm run compile      # コンパイル確認
# VSCode で F5 → Extension Development Host で動作確認
  2. バージョン番号を上げる — package.json の version フィールドを編集。SemVer 準拠で：
    • パッチ（バグ修正のみ）: 0.2.0 → 0.2.1
    • マイナー（後方互換ある機能追加）: 0.2.0 → 0.3.0
    • メジャー（互換性破壊）: 0.2.0 → 1.0.0

    npx --yes @vscode/vsce publish patch / minor / major でも自動加算できる。

  3. CHANGELOG.md に変更を追記 
    ## [0.2.1] - 2026-MM-DD

### Fixed
- 〇〇のときに××がうまく動かない問題を修正

### Changed
- △△の挙動を改善
  4. Git に commit してリモートに push 
    git add package.json CHANGELOG.md <変更したファイル>
git commit -m "release: v0.2.1 — 〇〇の修正"
git push origin main
  5. Marketplace に publish 
    npx --yes @vscode/vsce publish

    PAT が期限切れだとここで失敗する。その場合は PAT管理 を参照して再発行 → vsce login Maku でログイン後、再度 publish。

  6. 公開確認 — 2〜10分後に マーケットプレイスページ で新バージョンが反映されるか確認。VSCode の拡張機能タブからも更新が降ってくるはず。
git tag を打つ習慣 バージョンを上げたコミットに git tag v0.2.1 && git push origin v0.2.1 で タグを打っておくと、後で「v0.2.1 のソースを見たい」となったときに辿りやすい。GitHub の Releases 機能に紐付けて リリースノートを書くこともできる。

短縮フロー（vsce が自動で version up してくれる）

# patch / minor / major のいずれかを選んで
npx --yes @vscode/vsce publish patch
# → package.json の version を自動で +1 して commit して publish までやってくれる

ただし CHANGELOG への追記と git push は別途必要。CI/CD で自動化していない現状ではフルフローで進めるほうが安心。

PAT（Personal Access Token）の管理

現在の PAT の状況

  • WSL2 では keychain が動かないため、PAT は ~/.vsce にプレーンテキストで保存されている
  • 初回発行日：2026-05-18 / 有効期限：90日後（2026-08-16 頃失効）
  • スコープ：Marketplace → Manage
  • Organization：All accessible organizations
2026年12月1日以降の仕様変更 「All accessible organizations」スコープの PAT は廃止予定。それまでに別の認証方式（GitHub Actions 経由など）に 切り替えるか、または Marketplace 専用の認証手段に対応する必要がある可能性。詳細： Azure DevOps OAuth 等を参照。

PAT を再発行する

  1. PAT管理画面 を開く
  2. 右上「+ New Token」
  3. 下記設定で作成：
    Namevscode-publish-YYYYMM（年月を入れておくと管理しやすい）
    OrganizationAll accessible organizations
    Expiration90 days
    ScopesCustom defined → Show all scopes → Marketplace → Manage
  4. 表示されたトークンをコピー（一度しか見れない）
  5. ターミナルで再ログイン： 
    cd /home/osuim/Dev/DrawioInHtml
npx --yes @vscode/vsce login Maku
# プロンプトで PAT をペースト

PAT を破棄する（不要になったら / 漏れた疑いがあるとき）

  1. PAT管理画面 で該当トークンを選択 → Revoke
  2. ローカル保存も削除： 
    rm ~/.vsce
PAT が漏れたら即 Revoke PAT は「Marketplace の管理権限を持つ鍵」。漏れた場合は悪意ある拡張への差し替え等が可能。 少しでも疑わしければ Revoke して再発行する（コストはゼロ）。

Drawio Viewer (viewer-static.min.js) の更新

本拡張で使っている viewer-static.min.js は postinstall フックで https://viewer.diagrams.net/js/viewer-static.min.js から取得している（scripts/fetch-viewer.js）。

明示的に再取得して更新

rm media/viewer-static.min.js
npm run fetch-viewer
# → 最新の viewer-static.min.js が media/ に降りてくる

取得後、F5 で動作確認。問題なければ commit して publish。viewer 自体は media/ 配下にあり、 .gitignore されているので git 履歴には残らない。VSIX にだけ同梱される。

viewer の挙動が変わった場合 Drawio 側で API が変わると、現在の preview.js の createViewerForElement 呼び出しや 'check-visible-state': false 等の設定が効かなくなる可能性。 挙動異常を感じたら .claude/docs/knowledge/drawio-viewer-in-vscode-webview.md を再確認する。

Issue / バグ報告への対応

Issue の窓口

  • GitHub Issues：https://github.com/oSUiMiNo/DrawioInHtml/issues — 主要
  • Marketplace の Q&A：拡張機能ページの「Q&A」タブ — ユーザが直接質問する
  • Marketplace の Ratings & Reviews：星評価＋短いレビュー（こちらは技術的フィードバック向けではない）

対応のフロー

  1. 報告内容を再現する（環境差異、HTML サンプル、操作手順を確認）
  2. 原因が特定できたら GitHub Issue で進捗を返信
  3. 修正コードを書いて commit → 公開フロー に従って新バージョン publish
  4. CHANGELOG.md に修正内容を記録
  5. Issue に「fixed in vX.Y.Z」とコメントしてクローズ

緊急時：拡張機能の取り下げ・差し替え

バージョンを取り消したい

# 特定バージョンだけ取り消す
npx --yes @vscode/vsce unpublish Maku.drawio-in-html 0.2.0

既にインストール済みのユーザからは消えないが、新規ダウンロード不可になる。 代わりに修正版をすぐ publish するのが基本。

拡張機能ごと取り下げる

npx --yes @vscode/vsce unpublish Maku.drawio-in-html

Marketplace から完全に消える。ユーザにインストール済みの分は残るが、検索やインストール不可。 最終手段。同じ ID で再公開は将来できないかもしれないので慎重に。

Publisher ごと取り下げる（最終手段）

Publisher 管理ハブ から 「Delete publisher」。Publisher ID Maku は再利用不可になる可能性が高い。

不可逆操作 unpublish / delete publisher は一度実行すると基本的に元に戻せない。 「ちょっと様子を見て止めたい」程度なら、Marketplace 上の説明文に注意書きを追加する程度に留めるのが安全。

公開前チェックリスト

新バージョン公開前に確認すべき項目。一度ミスると数百ダウンロードが旧版にロックインされかねないので慎重に。

  • ☐ npm run compile がエラーなく通る
  • ☐ F5 で起動して sample/demo.html が想定通り表示される
  • ☐ 編集タブから保存して HTML に反映される
  • ☐ package.json の version を上げた
  • ☐ CHANGELOG.md に変更点を追記した
  • ☐ README.md が最新の機能仕様と一致している
  • ☐ Git で commit して git push origin main 済み
  • ☐ .vscodeignore に新たに不要なファイルが入っていないか
  • ☐ npx --yes @vscode/vsce ls --tree で同梱物が想定通り
  • ☐ PAT が有効期限内（不安なら再発行）

トラブルシューティング

publish 時に「The Personal Access Token verification has failed」

  • PAT が期限切れ → 再発行
  • PAT のスコープ不足 → Marketplace → Manage を含めて再発行
  • Organization が間違っている → All accessible organizations で再発行

publish 時に「Make sure to edit the README.md file before you publish」

  • テンプレ文字列（<your name here> 等）が残っている。README.md を確認して書き換える

publish 時に「Failed to publish: Extension validation failed」

  • package.json のフィールド漏れ（icon, repository, license 等）が原因のことが多い
  • エラーメッセージで指摘されている項目を直す

VSCode から「Maku.drawio-in-html」を検索しても出てこない

  • 公開直後はインデックスに時間がかかる（10〜30分待つ）
  • Ctrl+P → ext install Maku.drawio-in-html で ID 直接指定だと検索ヒットなしでもインストール可能

「Failed to open credential store」警告

  • WSL2 環境では keychain が動かないため、PAT は ~/.vsce にプレーンテキスト保存される
  • セキュリティ的に許容できないなら、毎回 vsce publish --pat <PAT> 形式で渡すか、CI/CD 経由に切り替える

viewer-static.min.js が postinstall で取れない

  • ネット接続を確認
  • https://viewer.diagrams.net/js/viewer-static.min.js をブラウザで直接ダウンロードして media/ に手動配置でも可
  • Drawio 側の URL が変わっていたら scripts/fetch-viewer.js の URL を書き換える
